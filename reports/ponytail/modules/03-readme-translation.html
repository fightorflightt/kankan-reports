原文

How it works — the Decision Ladder

Before an agent writes a single line, it walks this ladder top to bottom. The first rung that applies wins:

  1. Does this need to exist? → Skip it (YAGNI)
  2. Already in the codebase? → Reuse it
  3. In the standard library? → Use stdlib
  4. Platform / Browser native? → Use native (e.g. <input type="date">)
  5. Existing dependency can do it? → Use what's installed
  6. Can it be one line? → Write one line
  7. None of the above → Write the minimum viable code
中文

工作原理——决策阶梯

在 agent 写出任何一行代码之前，它必须从梯子顶端开始往下走。第一个满足条件的台阶就是它要做的：

  1. 这东西真的需要存在吗？ → 跳过（YAGNI 原则）
  2. 代码库里已经有了？ → 复用
  3. 标准库能搞定？ → 用标准库
  4. 平台/浏览器原生支持？ → 用原生功能（如 <input type="date">）
  5. 已经安装的依赖能解决？ → 用已有依赖
  6. 能一行写完吗？ → 写一行
  7. 以上都不行 → 写出最少能跑的代码
原文

Hard Red Lines — Never Cut

Ponytail is about removing unnecessary code, not necessary safety:

  • Input validation at trust boundaries
  • Error handling that prevents data loss
  • Security
  • Accessibility
中文

硬性红线——永远不能砍

Ponytail 砍的是不必要的代码，不是必要的安全措施：

  • 信任边界的输入校验
  • 防止数据丢失的错误处理
  • 安全性
  • 可访问性（Accessibility）
原文

Modes

Three intensity levels:

  • Lite — Gentle nudges, good for existing codebases with established patterns
  • Full (default) — Full ladder enforcement + aggressive simplification
  • Ultra — Every line must justify its existence. The agent will push back on you.
中文

模式

三种强度等级：

  • Lite（轻量） — 温和提醒，适合已有成熟模式的存量代码库
  • Full（默认） — 完整阶梯执行 + 激进简化
  • Ultra（极限） — 每一行代码都必须证明自己存在的理由。agent 会反过来质疑你的需求。
原文

Real-world results
