#AI 编程 #Agent Skill #工程规范 #开源

这是什么？

这是一套给「AI 编程助手」用的工作规矩手册。它把资深工程师做软件时的那套纪律——先想清楚要做什么、拆成小步、写测试证明能跑、评审过关再上线——写成 24 份说明书，逼着 AI 一步步照做，而不是图快走捷径。

打个比方：AI 写代码像个手很快但没经验的实习生，给它一个需求它就埋头开干，跳过文档、跳过测试、跳过检查。这个项目相当于给实习生配了一本「公司标准作业流程」，每做一步都要交作业、留证据，过不了关就不让往下走。作者是 Google Chrome 团队知名工程师 Addy Osmani。

应用场景

用 AI 认真写代码：让 AI 写代码时强制走『先规格、再测试、留证据』的工程流程，不图快走捷径
非技术 owner 兜底：不写代码的 owner + AI 写代码时，用它兜住工程质量
团队统一标准：给团队统一一套 AI 编码的纪律和评审标准

02

解决了什么痛点

为什么有人愿意花时间做这个项目？它回应了什么真实问题？

做这个项目的人，大概遇到了这些问题

用 AI 写代码的人，几乎都撞上过同一件事：AI 默认走最短路径。你让它做个功能，它会直接给你一坨「看起来能跑」的代码，但需求没问清、测试没写、边界没考虑、文档更别提。表面交付飞快，回头维护要命。

第二个痛点是「全凭感觉，没有证据」。AI 经常说「应该没问题」「这样改就对了」，但它不会主动去跑一遍、留下日志或测试结果来证明。等你发现不对，往往已经埋进生产环境了。

第三个痛点是AI 太会给自己找借口。"测试我待会儿再补"、"这个先简化一下"——这些听上去合理的偷懒话术，会让它一步步绕开本该做的事。靠人盯着一条条纠正，太累也不可持续。

说白了，作者想解决的是：AI 是个手很快、却没有「工程纪律本能」的实习生。那些不体现在代码改动里、却决定软件好坏的功夫（问清假设、控制改动大小、写规格、留证据、不让没法评审的东西合并），正是 AI 会自动跳过的部分。这套技能要做的，就是把这些纪律变成「AI 没法自己说服自己跳过」的硬规矩。

Want fewer manual steps once the spec exists? **/build auto** generates the plan and implements every task in a single approved pass — you approve the plan once, then it runs autonomously. It removes the human stepping *between* tasks, not the verification: every task is still test-driven and committed individually, and it pauses on failures or risky steps.

Skills also activate automatically based on what you're doing — designing an API triggers api-and-interface-design, building UI triggers frontend-ui-engineering, and so on.
中文

规格写好之后，想少点手动步骤？/build auto 会生成计划、并在一次获批的流程里实现每个任务——你只需批准计划一次，之后它就自主运行。它去掉的是任务之间需要人来推进的环节，而不是验证环节：每个任务依然是测试驱动、且各自单独提交，遇到失败或有风险的步骤会暂停。

Skill 也会根据你正在做的事情自动激活——设计 API 会触发 api-and-interface-design（API 与接口设计），构建 UI 会触发 frontend-ui-engineering（前端 UI 工程），以此类推。

### Meta
using-agent-skills — Maps incoming work to the right skill workflow and defines shared operating rules.

### Define
interview-me — One-question-at-a-time interview that extracts what the user actually wants, until ~95% confidence.
idea-refine — Structured divergent/convergent thinking to turn vague ideas into concrete proposals.
spec-driven-development — Write a PRD (objectives, commands, structure, code style, testing, boundaries) before any code.

### Plan
planning-and-task-breakdown — Decompose specs into small, verifiable tasks with acceptance criteria and dependency ordering.
中文

＃＃＃ Meta（元）——判断该用哪个 skill

using-agent-skills：把进来的活儿映射到正确的 skill 工作流，并定义共享的操作规则。适用于：开始一个会话、或要判断该用哪个 skill 时。

＃＃＃ Define（定义）——把要做什么搞清楚

interview-me：一次只问一个问题的访谈，挖出用户「真正想要」的（而非他自以为该要的），直到约 95% 的把握。适用于：需求没说清，或用户喊「interview me / grill me（拷问我）」时。

idea-refine：用结构化的发散/收敛思维，把模糊的想法变成具体的方案。适用于：你有个粗略概念、需要进一步探索时。

spec-driven-development：在写任何代码之前，先写一份 PRD，覆盖目标、命令、结构、代码风格、测试与边界。适用于：开新项目、新功能或重大改动时。

＃＃＃ Plan（计划）——拆解

planning-and-task-breakdown：把规格分解成一个个小而可验证的任务，带验收标准和依赖顺序。适用于：已有规格、需要拆成可落地单元时。

原文 
### Build
incremental-implementation — Thin vertical slices: implement, test, verify, commit. Feature flags, safe defaults, rollback-friendly.
test-driven-development — Red-Green-Refactor, test pyramid (80/15/5), DAMP over DRY, Beyonce Rule, browser testing.
context-engineering — Feed agents the right information at the right time: rules files, context packing, MCP integrations.
source-driven-development — Ground every framework decision in official docs; verify, cite sources, flag what's unverified.
doubt-driven-development — Adversarial fresh-context review of every non-trivial decision in-flight (CLAIM → EXTRACT → DOUBT → RECONCILE → STOP).
frontend-ui-engineering — Component architecture, design systems, state management, responsive design, WCAG 2.1 AA.
api-and-interface-design — Contract-first design, Hyrum's Law, One-Version Rule, error semantics, boundary validation.
中文

＃＃＃ Build（构建）——写代码

incremental-implementation：薄薄的「垂直切片」——实现、测试、验证、提交。配合 feature flag（功能开关）、安全的默认值、便于回滚的改动。适用于：任何改动超过一个文件时。

test-driven-development：红-绿-重构，测试金字塔（80/15/5），DAMP 优先于 DRY，Beyonce Rule（碧昂丝法则），浏览器测试。适用于：实现逻辑、修 bug 或改变行为时。

context-engineering：在对的时间把对的信息喂给 agent——rules 文件、上下文打包、MCP 集成。适用于：开始会话、切换任务、或输出质量下降时。

source-driven-development：每个框架决策都以官方文档为依据——核实、标注来源、把未经证实的内容标出来。适用于：你想要有出处、有权威依据的代码时。

doubt-driven-development：对每个进行中的非琐碎决策，用「全新上下文」做对抗式复审（声明 → 提取 → 质疑 → 调和 → 停止），并可选地授权跨模型升级核查。适用于：风险高（生产、安全、不可逆）、在陌生代码里、或现在核实比日后调试更划算时。

frontend-ui-engineering：组件架构、设计系统、状态管理、响应式设计、WCAG 2.1 AA 无障碍标准。适用于：构建或修改面向用户的界面时。

api-and-interface-design：契约优先设计、Hyrum's Law（海勒姆定律）、One-Version Rule（单版本规则）、错误语义、边界校验。适用于：设计 API、模块边界或公开接口时。

原文 
### Verify
browser-testing-with-devtools — Chrome DevTools MCP for live runtime data: DOM, console, network, performance profiling.
debugging-and-error-recovery — Five-step triage: reproduce, localize, reduce, fix, guard. Stop-the-line rule.

### Review
code-review-and-quality — Five-axis review, change sizing (~100 lines), severity labels (Nit/Optional/FYI).
code-simplification — Chesterton's Fence, Rule of 500, reduce complexity while preserving exact behavior.
security-and-hardening — OWASP Top 10 prevention, auth patterns, secrets management, dependency auditing.
performance-optimization — Measure-first: Core Web Vitals targets, profiling, bundle analysis, anti-pattern detection.
中文

＃＃＃ Verify（验证）——证明它能跑

browser-testing-with-devtools：用 Chrome DevTools MCP 获取实时运行数据——DOM 检查、控制台日志、网络追踪、性能剖析。适用于：构建或调试任何在浏览器里运行的东西时。

debugging-and-error-recovery：五步排查法——复现、定位、收敛、修复、加固。配合「停线（Stop-the-line）」规则与安全兜底。适用于：测试失败、构建中断或行为异常时。

＃＃＃ Review（评审）——合并前的质量关卡

code-review-and-quality：五维评审、改动控量（约 100 行）、严重度标签（Nit 小瑕疵 / Optional 可选 / FYI 知会）、评审速度规范、拆分策略。适用于：合并任何改动之前。

code-simplification：Chesterton's Fence（切斯特顿栅栏）、Rule of 500（500 行规则），在保持行为完全不变的前提下降低复杂度。适用于：代码能跑，但比它该有的样子更难读、更难维护时。

security-and-hardening：OWASP Top 10 防护、认证模式、密钥管理、依赖审计、三层边界体系。适用于：处理用户输入、认证、数据存储或外部集成时。

performance-optimization：测量优先——Core Web Vitals 指标、剖析工作流、打包体积分析、反模式检测。适用于：存在性能要求、或你怀疑出现性能退化时。

## How Skills Work
Every skill follows a consistent anatomy:

SKILL.md
  Frontmatter (name, description: Guides agents through [task]. Use when…)
  Overview         → What this skill does
  When to Use      → Triggering conditions
  Process          → Step-by-step workflow
  Rationalizations → Excuses + rebuttals
  Red Flags        → Signs something's wrong
  Verification     → Evidence requirements

**Key design choices:**
- Process, not prose. Skills are workflows agents follow, not reference docs they read.
- Anti-rationalization. Every skill includes a table of common excuses (e.g., "I'll add tests later") with counter-arguments.
- Verification is non-negotiable. Every skill ends with evidence requirements. "Seems right" is never sufficient.
- Progressive disclosure. The SKILL.md is the entry point. Supporting references load only when needed.
中文

＃＃ How Skills Work（skill 是怎么运作的）

每个 skill 都遵循一致的「解剖结构」：

一份 SKILL.md 包含：Frontmatter（开头的元信息：name 名称、description 描述——「引导 agent 完成〔某任务〕。在……时使用」）；Overview（概览）→ 这个 skill 做什么；When to Use（何时使用）→ 触发条件；Process（流程）→ 分步工作流；Rationalizations（借口）→ 常见借口及反驳；Red Flags（危险信号）→ 出问题的征兆；Verification（验证）→ 证据要求。

关键设计取舍：

· 是流程，不是散文。skill 是 agent 要照做的工作流，而不是供阅读的参考文档。每个都有步骤、检查点和退出标准。

· 反借口。每个 skill 都附一张表，列出 agent 用来跳过步骤的常见借口（例如「测试我待会儿再补」），并配上有据可依的反驳。

· 验证不可商量。每个 skill 都以证据要求收尾——测试通过、构建产物、运行数据。「看起来对」永远不够。

· 渐进式披露。SKILL.md 是入口；配套的参考资料只在需要时才加载，从而把 token 占用压到最低。

## Why Agent Skills?
AI coding agents default to the shortest path - which often means skipping specs, tests, security reviews, and the practices that make software reliable. Agent Skills gives agents structured workflows that enforce the same discipline senior engineers bring to production code.

Each skill encodes hard-won engineering judgment: when to write a spec, what to test, how to review, and when to ship. These aren't generic prompts - they're opinionated, process-driven workflows that separate production-quality work from prototype-quality work.

Skills bake in best practices from Google's engineering culture — including Software Engineering at Google and Google's engineering practices guide. You'll find Hyrum's Law in API design, the Beyonce Rule and test pyramid in testing, change sizing and review speed norms in code review, Chesterton's Fence in simplification, trunk-based development in git workflow, Shift Left and feature flags in CI/CD, and a dedicated deprecation skill treating code as a liability.

＃＃ Why Agent Skills?（为什么要 Agent Skills）

AI 编程 agent 默认走最短路径——这往往意味着跳过规格、测试、安全评审，以及那些让软件可靠的实践。Agent Skills 给 agent 提供结构化工作流，强制它们带着资深工程师对待生产代码的那种纪律。

创新点

和同类方案比，它有什么不一样？

和常规做法不一样的地方

市面上大多数「AI 提示词合集」给的是知识——告诉 AI「应该怎么做才对」。这个项目给的是流程和纪律——逼着 AI「必须按这个步骤做，做完还得交证据」。差别就像「发一本最佳实践手册」和「上一条带质检关卡的流水线」。

🚧
把「偷懒借口」提前写死

每个技能都附一张「反借口表」：把 AI 最爱说的偷懒话（"测试待会儿补""先简化一下"）连同反驳一起列出来。等于提前堵死后路，让它没法自己说服自己跳步。

✅
「拿证据」而不是「凭感觉」

每个技能都以「证据要求」收尾——必须给出测试通过、构建产物、运行数据。"看起来没问题"一律不算数。这把 AI 从「嘴上说行」拉到「证明给你看」。

🧩
用命令对应「研发全流程」

8 个命令（/spec→/plan→/build→/test→/review→/ship）正好串起从定义到上线的完整生命周期，而且会根据你在做什么自动激活对应技能——不用你记住该调哪个。

📦
渐进式加载，省「内存」

入口只是一份精简说明书，配套的详细清单「用到才加载」。对 AI 来说就是省 token、少占上下文；对产品人来说，这是一种「按需展开、不一次性塞爆」的信息架构思路。

🌍
一套内容，跨平台通用

技能本质是纯 Markdown 文本，所以能同时给 Claude Code、Cursor、Gemini、Copilot 等十来个工具用。一次编写，到处适配。

灵感卡片

看完这个项目我想到的产品点子——不一定用它的代码

灵感 1
给运营也配一套「反借口 SOP」

这个项目最妙的是「把常见偷懒话术连同反驳提前写死」。做单也一样：运营常说"图随便发一张就行""文案抄上次的"。可以做一份「做单标准动作 + 常见偷懒×正确做法」对照卡，直接发群里，比反复口头纠正有效得多。

灵感 2
「交证据才算完成」用在质检上

它要求每步都留证据、不准"看起来没问题"。可以套到代理/运营的工作验收：发单不是"发了就行"，而是要交「商品图+下单步骤图+群文案」三件套截图才算一单完成，后台自动记录。把模糊的"做了"变成可核对的"做对了"。

灵感 3
「按需展开」的帮助内容

它用「入口精简、细节用到才加载」省心。App 里的新人指引也该这样：首屏只放最关键的一句，"具体怎么做"折叠起来点了再看。别一上来甩一篇长教程，新代理根本读不完。

灵感 4
把"全流程"拆成几个一键入口

它用 6 个命令把研发全流程串起来，每步只管一件事。做单流程也能这么拆：选品 → 转链 → 做图 → 写文案 → 发群，每步一个清晰入口、做完自动带到下一步，而不是让运营在好几个页面之间来回跳。

灵感 5
一份内容、多端复用的内容策略

它"纯文本所以哪都能用"。我们的活动话术/做单模板也可以做成一份"源内容"，自动适配 App、H5、微信群不同场景的排版，而不是每个渠道各写一遍、还容易不一致。

产品化可行性

一个不会写代码的 PM，能不能用这个项目做出东西来？

低门槛
上手难度

一行命令装好就能用；本身是给「让 AI 帮你写代码」的人用的，自己不用懂代码

中等
依赖链

需要先有一个 AI 编程工具（Claude Code / Cursor 等），多数免费或低价；装插件要会用命令行

高适配
Vibe Coding 适配度

天生为「跟 AI 对话做开发」设计——它就是用来规范这种协作的，越用越省心

如果你要用，大概需要什么

对你（PM）来说，它不是拿来"集成进我们 App"的东西，而是你或团队里做小工具、小原型时的「研发纪律外挂」。前置条件很轻：装一个 AI 编程工具，跑一行安装命令。之后你用大白话提需求，它会逼着 AI 先帮你写清楚规格、再分步实现、每步留证据——相当于给你这个"非技术 owner"配了个会盯流程的资深工程师。

成本：时间上半天能上手；金钱上主要是 AI 工具本身的订阅费；人力上一个人就能跑。

⚠️
注意

它带来"纪律"也带来"慢"——会强制写规格、写测试、留证据，所以不适合"我就想 5 分钟糊个小 demo"的场景。它的价值在你真要把东西做扎实、要反复维护时才显现。另外，它管的是"过程规范"，不替你保证"想法对不对"——方向还得你来定。
