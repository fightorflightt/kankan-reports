#AI Agent #专家角色库 #跨工具 #开源

这是什么？

想象你开了一家「AI 人才外包公司」——232 个专业 AI 专家，从前端工程师到小红书运营、从安全架构师到 TikTok 策略师，覆盖 16 个部门。每个专家不是一句简单的提示词，而是有完整人设、工作流程、交付标准、成功指标的「数字员工档案」。

更狠的是：这套角色库不绑定任何一个 AI 工具。你可以一键安装到 Claude Code、Cursor、Copilot、Codex、Windsurf 等 18 种不同的 AI 编程工具里。117k Star 说明了一件事：「给我一个专业 AI 角色，而不是让我从头写 prompt」是爆发式需求。

02

解决了什么痛点

为什么 117,000 人给这个项目点了 Star？它回应了什么真实问题？

用 AI 干活的人，都在经历同样的困境

痛点一：「我每次都要重新写 prompt，而且写不好」。用 Claude 或 ChatGPT 干活时，大多数人的 prompt 就一句话：「帮我写个 React 组件」。AI 能写，但质量随机——有时候给的是过时的写法，有时候忽略了无障碍要求，有时候完全没考虑性能。而这个项目里的「Frontend Developer」Agent 有完整的身份设定、编码规范、测试策略、交付检查清单。你不需要懂怎么写 prompt，你只需要告诉它用哪个角色。

痛点二：「我换了工具，之前的 prompt 全废了」。今天用 Claude Code，明天团队换了 Cursor，后天老板说要用 Codex——每个工具的 agent 格式都不一样。The Agency 做了一件很聪明的事：同一套角色定义，通过一个 convert.sh 脚本自动转换成 18 种工具的格式。你只需要维护一套角色库，不用被工具锁定。

痛点三：「AI 像一个全能的实习生，但我要的是一个团队」。你需要的不是一个「什么都会一点」的通用 AI，而是一个前端工程师 + 一个安全审计员 + 一个 UI 设计师 + 一个增长黑客，各自在自己的领域是高手。232 个角色覆盖了软件开发从写代码到做营销的全链条——这本质上是把「一个人的 AI 助手」升级成了「一个 AI 公司」。

04

创新点

和常规的「Prompt 模板库」比，它有什么本质不同？

这不是 prompt 合集，这是一套「AI 人才操作系统」

GitHub 上有无数个「Awesome ChatGPT Prompts」类的仓库。The Agency 做出了本质差异：它把 prompt 升级成了 agent——不是一段话，而是一个有身份、有流程、有交付标准的可安装文件。

🔄
工具无关的跨平台转换引擎

这是最核心的创新。大多数 agent 库只绑定一个工具，而它通过 convert.sh 脚本将同一套 agent 定义映射到 18 种工具的本地格式。这相当于「一次编写，到处运行」——你的 AI 团队不随工具迁移而丢失。

🏢
「完整公司」而不是「单一助手」

232 个角色覆盖了从写代码（工程部）到做设计（设计部）到搞营销（营销部）到管安全（安全部）到做游戏（游戏开发部）的全链条。别的 prompt 库给你一个「万能助手」，它给你一整家公司——你可以按部门选人、按项目组队。

📋
像招聘一样的 agent 设计

每个 agent 文件都有：身份与性格、核心使命、工作流程、交付物标准、沟通风格。这不像写 prompt，更像在写一份职位描述（JD）。用户可以像「面试」一样选择一个 agent：看它的专长、看它过往的交付标准、看它适不适合当前项目。

🇨🇳
深度的中国市场覆盖

极其少见的一点：232 个角色中包含了微信小程序开发者、小红书运营、飞书集成开发、知乎策略师、百度 SEO、抖音/B站/快手策略师等大量中国市场专属角色。这不是一个「翻译」出来的仓库，而是有真实中国市场场景思考的全球化产品。中文 fork 版（agency-agents-zh）更是增加了 50 个中国市场原创 agent。

05

灵感卡片

看完这个项目我想到的产品点子——不一定用它的代码

灵感 1
「AI 角色市场」——像 App Store 一样分发 Agent

117k Star 证明了对预定义 AI 角色的需求。可以做一个 agent 市场：用户搜场景（如「做小红书运营」）→ 找到对应 agent → 一键安装到自己的 AI 工具 → 评价和评分。本质是「AI 时代的技能市场」，每个 agent 都是一个可交易的数字商品。

灵感 2
「团队编排器」——多 Agent 协作的主控台

232 个 agent 是原材料，但用户还需要一个「项目经理」来协调它们。可以做一个编排层：用户描述一个任务（如「帮我做一个电商 App」），系统自动选出需要的 agent（前端 + 后端 + UI 设计 + 支付集成 + 安全审计），按 DAG 顺序调度它们协作。仓库里已有的 agents-orchestrator 和 multi-agent-systems-architect 说明作者也在往这个方向想。

灵感 3
Agent 版本管理和 A/B 测试

这个项目本质上是一套 agent 的「源码仓库」。既然是代码，就需要版本管理、变更追踪、质量回归。可以做一个 agent CI/CD 平台：当你修改一个 agent 的定义时，自动跑一组测试用例（同样的输入，新旧 agent 分别输出什么？哪个更好？），用数据而非感觉来判断 agent 改进了还是退化了。

灵感 4
非技术场景的 Agent 角色扩展

这个仓库虽然偏向技术/营销场景，但框架可以套到任何领域。比如律师行业的 agent 集合（合同审核 agent + 诉讼策略 agent + 合规检查 agent）、医疗行业（诊断辅助 agent + 病历整理 agent + 患者沟通 agent）、教育行业（课程设计 agent + 作业批改 agent + 学习路径规划 agent）。谁先做了某个垂直行业的 agent 全家桶，谁就吃了第一波红利。

灵感 5
「Agent 即服务」的 SaaS 模式

企业不需要维护 232 个 agent 文件——他们只需要在需要的时候调用对的 agent。可以把这个仓库打包成 API 服务：客户传一个需求描述 → API 自动匹配 agent → agent 执行 → 返回结果。按调用次数或订阅收费。这比卖 agent 文件更高频、更可计量、更容易变现。

06

产品化可行性

一个不会写代码的 PM，能不能用这个项目做出东西来？

低门槛
直接使用

如果你在用 Claude Code，一行命令安装：./scripts/install.sh --tool claude-code。然后直接说「激活 Frontend Developer 帮我写组件」就行。

中等
依赖链

需要先有一个支持的 AI 编程工具（Claude Code / Cursor / Codex 等）。clone 仓库后运行安装脚本即可。不需要额外付费。

非常适合
Vibe Coding 适配度

不仅能用，而且是 vibe coding 的加速器——你不需要自己写 prompt 来让 AI 做好前端开发，直接用「Frontend Developer」agent 就行。对 PM 来说，这降低了让 AI 产出高质量代码的门槛。

如果你要基于这个项目做产品，大概需要什么

• 做 agent 市场 / 编排平台：需要开发团队，2-3 个工程师 + 1 个设计师，3-6 个月 MVP。核心难点在编排引擎和跨工具兼容性。

• 做垂直行业的 agent 全家桶：1 个 PM（你）+ Claude Code 写 agent 定义文件，2-4 周能做第一版（10-20 个角色）。门槛极低，但需要行业知识。

• 纯用这个仓库提效：零成本，今天就能装。最大的投入是熟悉各个 agent 能做什么。

⚠️
注意

质量参差不齐：232 个 agent 来自社区贡献，质量必然有高有低。有些 agent 定义详尽、流程清晰；有些可能只是泛泛的 prompt。需要试用筛选。

工具的 agent 能力上限：agent 的定义写得再好，最终效果取决于底层 AI 工具的能力。同一个「Security Architect」agent 在 Claude Code（Opus 4.8）和 Kimi Code 上的表现可能天差地别。

维护是个体力活：如果你 fork 了想长期维护，需要跟踪 18 种工具的 agent 格式变化、更新 232 个角色定义、处理社区 PR——这不是一个小工程。

07

同类项目速览

在 AI Agent 角色库的生态里，它处于什么位置

Cursor Rules 生态 Cursor 官方的 .cursorrules 格式，偏 IDE 内规则（代码风格、提交规范），覆盖范围窄。The Agency 覆盖了从工程到营销的全链条，且支持 18 种工具。但 Cursor Rules 在 IDE 内的体验更原生。
awesome-chatgpt-prompts (110k+ Stars) 最经典的 prompt 合集，但每个角色只是一段文字描述，没有流程、没有交付标准、不支持工具安装。The Agency 本质上是对这个模式的升级：从「一段话」变成「一个可安装的专家」。
jnMetaCode/agency-agents-zh (15.4k Stars) The Agency 的中文 fork 版，266 个 agent（多出 50 个中国市场原创角色）。搭配了 agency-orchestrator 编排器，支持 DAG 多 agent 协作。是中文用户的首选版本，但跨工具兼容性不如原版。
Claude Code 官方 Skills 市场 Claude Code 内置的 skill/agent 机制，MCP 协议加持，能力更强（可以调外部 API），但角色数量远不及 The Agency。两者互补：The Agency 提供角色灵感，Claude Code 提供执行环境。
OpenClaw Skills 生态 OpenClaw 的 Skill 机制（SOUL.md + SKILL.md）和 The Agency 的 agent 定义在思路上高度相似——都是 Markdown 文件定义角色。但 OpenClaw 偏向 24/7 自动化场景，The Agency 偏向项目协作场景。
📌
这个项目适合：

• 想让 AI 在不同场景「换角色」干活的人——写代码用工程师、做设计用设计师、搞营销用增长黑客

• 在多工具间切换的团队——统一用一套角色定义，不被工具锁定

• 想做 agent 相关产品的 PM——这是最好的「agent 应该怎么设计」的学习素材


不适合：

• 追求极致性能的场景——agent 定义再好，效果最终还是看底层模型和工具能力

• 只用一种工具且已有成熟 prompt 库的人——直接装 232 个 agent 反而是噪音

• 需要实时数据/API 调用的自动化场景——这些 agent 是角色定义，不包含 MCP tool 能力
