#AI 私教 #个性化学习 #长期陪伴 #Claude Code Skill

这是什么？

这不是一个 App，而是 Matt Pocock 那套 skills 合集里的一个单独的 skill——「teach」。它是一份「给 AI 看的家教说明书」：你对 AI 说「教我学 X」，它就不再是随便答两句，而是按这份说明书，把自己变成一个有记性、会因材施教、能陪你跨越好多次的私人家教。

关键在「有记性」三个字。普通和 AI 聊天，关掉就忘了。这个 skill 让 AI 把你的学习状态——你为什么学、学到哪了、踩过哪些坑、你的偏好——全都记在一个文件夹里，下次接着教。说白了，它把「一次性问答」升级成了一段有进度条的师生关系。

应用场景

系统自学一门硬技能：比如想学产品数据分析、SQL、英语写作。它会先问清你「到底为啥学」，再按你的水平一节一节地教，每节都短小、有练习、有反馈，而不是甩给你一堆资料。
带明确目标的速成：比如「10 月要跑半马」「下周要给团队讲清楚某个概念」。它要求你把目标定得很具体，然后所有课程都围绕这个目标，不跑题、不灌无关知识。
非技术领域也能用：作者特意举了瑜伽、健身的例子——它不只教知识，也能带你做「真实世界的动作清单」（一个个瑜伽体式），并提醒你去线下社群找真人反馈。
沉淀一本「自己的教材」：学的过程中它会帮你生成精美的速查卡、术语表、学习记录。学完你手里不是一堆聊天记录，而是一套可反复翻阅、还能打印的个人参考资料。
02

解决了什么痛点

为什么要专门写一个「教学」skill？普通问 AI 不行吗？

用 AI 学东西的人，通常会撞上这三堵墙

第一堵墙：AI 没记性，每次都从零开始。你今天问 AI 学 SQL，它讲一通；明天再来，它完全不记得你昨天学到哪、哪里没懂。学习本该是「层层往上盖」的，但和 AI 的对话每次都被推倒重来。这个 skill 的解法是：把你的学习状态（为什么学、学到哪、踩过哪些坑）全部写进一个文件夹长期保存，让 AI「跨天还认识你」。

第二堵墙：AI 答得太多、太泛，像在背百科。你随便问一句，它能给你一大段，但既不知道你的真实目标，也不管你现在是什么水平，结果就是「看着都懂、其实没学会」。这个 skill 强制 AI 先搞清楚你「到底为了什么而学」（它叫 Mission/使命），并且每节课只教「刚好比你现在难一点点」的内容——既不让你觉得无聊，也不让你直接劝退。

第三堵墙：看着像学会了，其实记不住。这是学习里最隐蔽的坑——作者专门点名：「当下能想起来」（流畅度）会给你一种「我掌握了」的错觉，但真正的目标是「长期记得住」（存储强度）。普通 AI 问答只给你前者。这个 skill 刻意用「回忆测验、间隔重复、交叉练习」这些有科学依据的方法，制造「适度的困难」，逼着知识真正沉进长期记忆。

用户请你教他们点东西。这是一个「有状态的」（stateful）请求——他们打算跨越多次会话来学习这个主题。（「有状态」= 有记忆、有进度，不是一次性的。）

原文

## Teaching Workspace

Treat the current directory as a teaching workspace. The state of their learning is captured in this directory in several files:

中文

## 教学工作区（Teaching Workspace）

把当前目录当作一个教学工作区。用户的学习状态被记录在这个目录的若干文件里：

原文
  • MISSION.md: A document capturing the reason the user is interested in the topic. This should be used to ground all teaching.
  • ./reference/*.html: A directory of reference materials... cheat sheets, reference algorithms, syntax, yoga poses, glossaries... They should be beautiful documents which print out well, and are designed for quick reference.
  • RESOURCES.md: A list of resources which can be explored to ground your teaching in contextual knowledge, or to acquire knowledge and wisdom.
中文
  • MISSION.md：记录用户对这个主题感兴趣的原因的文档。所有教学都应以它为根基。
  • ./reference/*.html：参考资料目录……速查表、参考算法、语法、瑜伽体式、术语表……它们应该是排版精美、打印出来也好看、为快速查阅而设计的文档。
  • RESOURCES.md：一份可供探索的资源清单，用来把你的教学扎根在情境知识里，或用来获取知识与智慧。
原文
  • ./learning-records/*.md: A directory of learning records, which capture what the user has learned... loosely equivalent to architectural decision records in software development... titled 0001-<dash-case-name>.md, where the number increments each time.
  • ./lessons/*.html: A directory of lessons. A lesson is a single, self-contained HTML output that teaches one tightly-scoped thing tied to the mission.
  • ./assets/*: Reusable components shared across lessons.
  • NOTES.md: A scratchpad for you to jot down user preferences, or working notes.
中文
  • ./learning-records/*.md：学习记录目录，记录用户已经学会了什么……大致相当于软件开发里的「架构决策记录」（ADR）……命名为 0001-<短横线命名>.md，编号每次递增。
  • ./lessons/*.html：课程目录。一节 lesson（课） 是一个独立完整的 HTML 输出，只教一件与 mission 紧密相关、范围很窄的事。
  • ./assets/*：跨课程复用的 components（组件）。
  • NOTES.md：一个草稿本，供你随手记下用户偏好或工作笔记。
原文

## Philosophy

To learn at a deep level, the user needs three things:

  • Knowledge, captured from high-quality, high-trust resources
  • Skills, acquired through highly-relevant interactive lessons devised by you, based on the knowledge
  • Wisdom, which comes from interacting with other learners and practitioners
中文

## 理念（Philosophy）

要深度学习，用户需要三样东西：

  • 知识（Knowledge），从高质量、高可信度的资源中获取
  • 技能（Skills），通过你基于知识设计的、高度相关的互动课程习得
  • 智慧（Wisdom），来自与其他学习者和从业者的互动
原文

Before the RESOURCES.md is well-populated, your focus should be to find high-quality resources which will help the user acquire knowledge. Never trust your parametric knowledge.

Some topics may require more skills than knowledge. Learning more about theoretical physics might be more knowledge-based. For yoga, more skills-based.

中文

在 RESOURCES.md 还没填充充分之前，你的重点应该是找到能帮用户获取知识的高质量资源。永远不要相信你自己脑子里的（parametric）知识。（即：别凭 AI 自己的记忆瞎讲，要去找可靠出处。）

有些主题需要的技能多于知识。学理论物理可能更偏知识；学瑜伽则更偏技能。

原文

### Fluency vs Storage Strength

You should be careful to split between two types of learning:

  • Fluency strength: in-the-moment retrieval of knowledge
  • Storage strength: long-term retention of knowledge
中文

### 流畅度 vs 存储强度（Fluency vs Storage Strength）

你应当小心区分两种学习：

  • Fluency strength（流畅度）：当下、即时地想起某个知识
  • Storage strength（存储强度）：知识的长期留存
原文

Fluency can give the user an illusory sense of mastery, but storage strength is the real goal. Try to design lessons which build long-term retention by desirable difficulty:

  • Using retrieval practice (recall from memory)
  • Spacing (distributing practice over time)
  • Interleaving (mixing up different but related topics in practice - for skills practice only)
中文

流畅度会给用户一种「我掌握了」的错觉，但存储强度才是真正的目标。要努力设计能通过「适度的困难」（desirable difficulty）建立长期留存的课程：

  • 使用 retrieval practice（回忆练习——从记忆中主动回想）
  • Spacing（间隔——把练习分散在不同时间）
  • Interleaving（交叉——在练习中混入不同但相关的主题，仅用于技能练习）
原文

## Lessons

A lesson is the main thing you produce — the unit in which knowledge and skills reach the user. Each lesson is one self-contained HTML file, saved to ./lessons/ and titled 0001-<dash-case-name>.html where the number increments each time.

A lesson should be beautiful — clean, readable typography and layout — since the user will return to these later to review. Think Tufte.

中文

## 课程（Lessons）

课，是你产出的主要东西——是知识与技能抵达用户的「单位」。每节课是一个独立完整的 HTML 文件，存到 ./lessons/，命名为 0001-<短横线命名>.html，编号每次递增。

一节课应该是精美的——干净、易读的排版与布局——因为用户之后会回来复习。参照 Tufte 的风格。（Edward Tufte，信息设计大师，以极简优雅著称。）

原文

The lesson should be short, and completable very quickly. Learners' working memory is very small, and we need to stay within it. But each lesson should give the user a single tangible win that they can build on. It should be directly tied to the mission, and should be in the user's zone of proximal development.

If possible, open the lesson file for the user by running a CLI command.

中文

课程应该短、能很快做完。学习者的「工作记忆」（working memory，即当下能同时处理的信息量）非常小，我们必须待在它的容量之内。但每节课都应给用户一个可以继续往上搭的、看得见摸得着的小胜利。它应直接与 mission 挂钩，并落在用户的 zone of proximal development（最近发展区）之内。

如果可能，通过运行一条 CLI 命令帮用户把课程文件打开。

原文

Each lesson should link via HTML anchors to other lessons and reference documents.

Each lesson should recommend a primary source for the user to read or watch. This should be the most high-quality, high-trust resource you found on the topic.

Each lesson should contain a reminder to ask followup questions to the agent. The agent is their teacher, and can assist with anything that's unclear.

中文

每节课都应通过 HTML 锚点链接到其他课程和参考文档。

每节课都应推荐一个 primary source（首要来源）供用户阅读或观看。这应是你就该主题找到的最高质量、最可信的资源。

每节课都应包含一句提醒：可以向 agent 追问后续问题。Agent 就是他们的老师，能帮忙解答任何不清楚的地方。

原文

## Assets

Lessons are built from reusable components, stored in ./assets/: stylesheets, quiz widgets, simulators, diagram helpers — anything a second lesson could reuse.

Reuse is the default, not the exception... When a lesson needs something new and reusable, write it as a component in ./assets/ and link to it — never inline code a future lesson would duplicate.

中文

## 资产（Assets）

课程由可复用的 components（组件） 搭成，存放在 ./assets/：样式表、测验小部件、模拟器、图表助手——任何第二节课可能复用的东西。

复用是默认，而非例外……当一节课需要某个新的、可复用的东西时，把它写成 ./assets/ 里的组件并链接过去——绝不把以后会重复的代码内联写死在一节课里。

原文

A shared stylesheet is the first component every workspace earns: every lesson links it, so the lessons look like one consistent course rather than a pile of one-offs. As the workspace grows, so should the component library.

中文

共享样式表是每个工作区应该挣到的第一个组件：每节课都链接它，于是这些课看起来像「一门统一的课程」，而不是一堆各做各的散件。随着工作区成长，组件库也应随之成长。

原文

## The Mission

Every lesson should be tied into the mission - the reason that the user is interested in learning about the topic.

If the user is unclear about the mission, or the MISSION.md is not populated, your first job should be to question the user on why they want to learn this.

中文

## 使命（The Mission）

每节课都应与 mission 绑定——也就是用户对这个主题感兴趣的原因。

如果用户对 mission 不清楚，或者 MISSION.md 还没填，你的第一项工作就是盘问用户「为什么想学这个」。

原文

Failing to understand the mission will mean knowledge acquisition is not grounded in real-world goals. Lessons will feel too abstract...

Missions may change as the user develops more skills and knowledge. This is normal... Confirm with the user before changing the mission.

中文

没搞懂 mission，就意味着知识获取没有扎根在真实世界的目标上。课程会显得太抽象……

随着用户技能和知识增长，mission 可能会变。这很正常……在改动 mission 之前，先和用户确认。

原文

## Zone Of Proximal Development

Each lesson, the user should always feel as if they are being challenged 'just enough'.

The user may specify an exact thing they want to learn. If they don't, figure out their zone of proximal development by: reading their learning-records; figuring out the right thing to teach based on their mission; teach the most relevant thing that fits in their zone of proximal development.

中文

## 最近发展区（Zone Of Proximal Development）

每一节课，用户都应始终感觉自己被「恰到好处」地挑战着。（最近发展区：比你现有水平略高一点、跳一跳够得着的难度区间。）

用户可能会指定他们想学的确切内容。如果没有，就这样推算他们的最近发展区：读他们的 learning-records；根据他们的 mission 判断该教什么；教那个最相关、又落在最近发展区里的内容。

原文

## Knowledge

Lessons should be designed around a skill the user is going to learn. The knowledge in the lesson should be only what's required to acquire that skill. You teach the knowledge first, then get the user to practice the skills via an interactive feedback loop.

Knowledge should first be gathered from trusted resources... Lessons should be littered with citations... For acquiring knowledge, difficulty is the enemy. It eats working memory you need for understanding.

中文

## 知识（Knowledge）

课程应围绕用户即将学习的某项技能来设计。课里的知识应该只包含习得该技能所必需的部分。你先教知识，再通过互动的反馈回路让用户练习技能。

知识应首先从可信资源中收集……课程里应当遍布引用（citations）……对于获取知识，难度是敌人。它会吃掉你理解所需的工作记忆。

原文

## Skills

If knowledge is all about acquisition, skills are about durability and flexibility. Make the knowledge stick.

For skill acquisition, difficulty is the tool. Effortful retrieval is what builds storage strength. Skills should be taught through interactive lessons. There are several tools at your disposal: interactive lessons, using quizzes and light in-browser tasks; lessons which guide the user through a list of real-world steps to take (for instance, yoga poses).

中文

## 技能（Skills）

如果说知识关乎「获取」，那么技能关乎「持久与灵活」。让知识真正粘住。

对于技能习得，难度是工具。费力的回忆才能建立存储强度。技能应通过互动课程来教。你有几样工具可用：用测验和轻量的浏览器内任务做成的互动课；引导用户走完一串「真实世界动作清单」的课（比如瑜伽体式）。（注意这里和「知识」相反：学知识时难度是敌人，练技能时难度是工具。）

原文

Each of these should be based on a feedback loop... This feedback loop should be as tight as possible, giving feedback immediately - and ideally automatically.

For quizzes, each answer should be exactly the same number of words (and characters, if possible). Don't give the user any clues about the answer through formatting.

中文

这些都应建立在一个反馈回路（feedback loop）之上……这个反馈回路应尽可能「紧」，立刻给出反馈——最好是自动的。

对于测验，每个选项的字数（以及字符数，如果可能）都应完全一致。不要通过排版给用户任何关于正确答案的暗示。（防止靠「哪个选项更长」蒙对。）

原文

## Acquiring Wisdom

Wisdom comes from true real-world interaction... When the user asks a question that appears to require wisdom, your default posture should be to attempt to answer - but to ultimately delegate to a community.

A community is a place (online or offline) where the user can test their skills... You should attempt to find high-reputation communities... If the user expresses a preference that they don't want to join a community, respect it.

中文

## 获取智慧（Acquiring Wisdom）

智慧来自真实世界的互动……当用户问一个看起来需要「智慧」才能回答的问题时，你的默认姿态应该是：试着回答——但最终把它交给一个社群（community）。

社群是一个（线上或线下的）让用户能在真实世界检验自己技能的地方……你应努力找到高声誉的社群……如果用户表示不想加入社群，尊重这个意愿。

原文

## Reference Documents

While creating lessons, you should also create reference documents... Lessons will rarely be revisited later - reference documents will be. They should be the compressed essence of the lesson, in a format designed for quick reference.

中文

## 参考文档（Reference Documents）

在创建课程的同时，你也应创建参考文档……课程很少会被回头重看——参考文档会。它们应是课程的「压缩精华」，采用为快速查阅而设计的格式。

原文

Some learning topics lend themselves to reference: Syntax and code snippets for programming; Algorithms and flowcharts for processes; Yoga poses and sequences for yoga; Exercises and routines for fitness; Glossaries for any topic with its own nomenclature.

Glossaries, in particular, are an essential reference. Once one is created, it should be adhered to in every lesson.

中文

有些学习主题天然适合做成参考：编程的语法和代码片段；流程的算法和流程图；瑜伽的体式和序列；健身的训练动作和计划；任何有自己一套术语体系的主题的术语表（Glossary）。

尤其是术语表，是一种必备的参考。一旦建立，每节课都应遵守它。

原文

## NOTES.md

The user will sometimes express preferences of how they want to be taught, or things you should keep in mind. This is the place to record those preferences, so you can refer back to them when designing lessons or working with the user.

中文

## NOTES.md（笔记）

用户有时会表达他们希望被怎么教，或一些你该记住的事。这里就是记录这些偏好的地方，方便你在设计课程或与用户协作时回头参考。

原文

（另：teach 文件夹里还有 4 个「格式规范」文件，规定上面几个状态文件长什么样）

  • MISSION-FORMAT.md
  • RESOURCES-FORMAT.md
  • LEARNING-RECORD-FORMAT.md
  • GLOSSARY-FORMAT.md
中文

这 4 个文件是给 AI 的「填表模板」，要点摘译——

  • MISSION 规范：一个工作区只能有一个 mission；目标要具体（「10 月跑半马」胜过「变健康」）；用户说不清就先访谈再写；目标变了就更新。
  • RESOURCES 规范：只收高可信来源；每条都要写一句「它讲什么、什么时候该用它」；分「知识 / 智慧（社群）」两组；缺好资源就显式标注缺口。
  • LEARNING-RECORD 规范：像软件的 ADR，只记「非显而易见的洞察 / 用户已掌握的 / 被纠正的误解 / 目标的转变」，不是流水账日记；理解更新了就把旧记录标为「已被取代」而非删除。
  • GLOSSARY 规范：只有当用户真正理解一个词，才把它加进术语表；定义要短、要有主见（一个概念多个叫法时只选最好的一个）；定义里也尽量用术语表自己的词。
04

创新点

和「直接问 AI」「上网课」比，它特别在哪？

和常规做法不一样的地方

它最聪明的一点：不靠 AI 的「记忆力」，而是靠一个文件夹当「记忆外挂」。AI 本身记不住跨天的事，但这个 skill 把你的目标、进度、踩过的坑、术语表全写成文件存下来，AI 每次开课先读一遍——于是一个本来「健忘」的 AI，变成了一个「认识你、知道你学到哪」的长期家教。它把成熟的学习科学（最近发展区、间隔重复、流畅度 vs 存储强度）直接写进了给 AI 的指令里。

🗂️
用文件夹做「记忆外挂」

MISSION / 学习记录 / 术语表 / 笔记，几个文件就把「学习状态」持久化下来。这是它能跨多次会话陪你的根本，也是它和「一次性问答」的本质区别。

🎯
先逼问「你到底为啥学」

没搞清 Mission 不准开课。这一招直接借用了同仓库 grill（盘问）的思路：把学习扎根在真实目标上，避免「学了一堆用不上的抽象知识」。

🧠
把「学习科学」写进指令

明确区分「当下想得起」和「长期记得住」，并刻意制造「适度困难」（回忆测验、间隔、交叉练习）。还有个反直觉规矩：学知识时难度是敌人，练技能时难度是工具。

📐
课程「小而美、可复用」

每节课短到几分钟做完（迁就人的工作记忆有限），排版要精美（学完能当资料回看），还共享一套样式和组件，让零散的课看起来像「一门完整的课」。

🤝
承认 AI 教不了「智慧」

难得的清醒：知识和技能 AI 能教，但「智慧」要去真实社群里练。它会主动帮你找靠谱的论坛/线下班，而不是假装自己什么都能教——给 AI 的能力边界划了条诚实的线。

05

灵感卡片

看完这个 skill 我想到的产品点子——不一定用它的代码

灵感 1
「带记忆」的新代理培训陪练

这个 skill 最值钱的点是「用文件记住每个学习者的进度」。搬到咱们培训新代理：给每个新代理建一份「成长档案」（他卡在哪、哪类商品不会做单、上次练到哪），AI 培训助手每次接着上次教，而不是每个新人都从头讲一遍同样的话。把「一次性培训」变成「有进度条的陪练」。

灵感 2
先问「你要冲什么单」再教做单

它「不搞清 Mission 不开课」的规矩很值得借。给代理的做单辅导别一上来就教技巧，先问清「你这个群是什么人、这次想冲量还是冲客单价」，再针对性教。让辅导扎根在代理的真实目标上，而不是给一套通用模板。

灵感 3
「学完留下一套精美资料」的内容思路

它强调每节课要做成「精美、能回看、能打印」的资料，而不是聊完就散。咱们做代理培训内容/用户教程时也可以这样：每个知识点产出一张设计精良的「速查卡」（怎么发新品、怎么追单话术），代理能存能转发能随时翻，比一大段文字或一段没人回看的语音强得多。

灵感 4
「最近发展区」式的难度分层

它每次只教「比你现在难一点点」的内容。咱们给代理的成长路径也可以这样设计：别把所有玩法一次倒给新人，按「他现在的水平」推下一个刚好够得着的小目标（先学会发单 → 再学会配图 → 再学会做下单步骤视频），一步一个小胜利，留存更好。

灵感 5
把「知识 → 技能 → 智慧」用在运营成长上

它把学习拆成三层：知识（看资料）、技能（练 + 即时反馈）、智慧（去真人社群历练）。对应到代理成长就是：看 SOP（知识）→ 实操做单拿反馈（技能）→ 进优秀代理交流群偷师（智慧）。这个三层框架能帮你想清楚「光发教程为什么不够」——缺的是后两层。

06

产品化可行性

一个不会写代码的 PM，能不能自己用上这个 skill？

低门槛
上手难度

它就是一份说明书，没有要装的软件。在 Claude Code 里把这个 skill 装上，输入 /teach 我想学 X 就开始，全程像聊天。

中等
依赖链

前提是你得在电脑上用着 Claude Code（或兼容的 agent），并愿意让它在一个文件夹里读写文件。对纯手机用户是门槛。

低门槛
Vibe Coding 适配度

天生为「对话式」而生，用法就是和 AI 聊。你不需要懂任何代码，它产出的课程是网页，双击就能看。

如果你要用，大概需要什么

对你个人学习来说，这东西基本是「拿来即用」：一台装了 Claude Code 的电脑、从 Matt 的 skills 合集里把 teach 装上、给它指一个空文件夹当「学习工作区」，然后 /teach 我想学 ⋯⋯。不花钱（MIT，免费）。真正的成本是 AI 的使用额度，以及你愿不愿意按它的节奏（先被盘问目标、一节节慢慢来）学。它特别适合你这种「自己也做独立小产品、想系统补个技能」的人。

但如果你想的是「把它做成给代理/用户的学习产品」，那它不能直接搬——它是给「会用 Claude Code 的人」的工具，不是给手机里的运营。对团队场景，它的价值在那套教学方法论（见灵感卡片），而非现成可发的产品。

⚠️
注意

① 它的「记忆」全靠那个文件夹——文件夹丢了/换了，进度就没了，得固定用同一个；② 一个工作区只能学一个主题，想学两样要开两个文件夹；③ 它要求 AI「别凭记忆瞎讲、要去找可靠资源」，所以质量取决于 AI 联网找资料的能力，遇到冷门主题可能资源不足；④ 它是一个人维护的个人 skill，风格很「私人」、更新快，别当成稳定产品。

07

同类项目速览

想「让 AI 教我学东西」，还有哪些路子？它处在什么位置

直接问 ChatGPT / Claude 最普遍的做法。胜在零门槛、即问即答；输在没记性、不成体系、容易给你「学会了的错觉」。teach 正是冲着补这些短板来的——给「随便聊」加上记忆、结构和学习科学。
Duolingo / 多邻国这类 App 成品学习 App，体验打磨好、有游戏化和间隔重复。但只覆盖固定科目（语言等），内容是给所有人的标准课。teach 反过来：科目无限、完全按你个人目标定制，但需要你自己会用 AI 工具、且没有现成精美 App 的丝滑。
Khan Academy / 各类网课 高质量人类讲师、体系完整。但是「一对多」的标准课，不会围着你一个人的目标转，也不会记住你的进度。teach 的卖点恰是「一对一、围着你的 Mission 转」。
同仓库的 grill / handoff 等 skill teach 其实复用了同仓库「盘问（grill）」和「文件持久化记忆（类似 handoff 交接）」的思路，是这套方法论在「学习」场景下的一次组合应用。想了解整套合集可看本站另一篇 mattpocock/skills 报告。
📌
这个 skill 适合：

已经在用 Claude Code、想系统自学某个具体技能、且看重「个性化 + 长期进度 + 留下可复习资料」的人（比如你这种自己做独立产品、需要随时补技能的 PM）。


不适合：

① 只想快速查个答案、不打算长期学的人——直接问 AI 更省事；② 完全不碰电脑工具的手机用户；③ 想要现成丝滑学习 App 体验、不愿自己折腾文件夹的人。
