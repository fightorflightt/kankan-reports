这是什么？

这不是一个软件，而是一本 42 万字的免费在线技术书。它干的事只有一件：把 Claude Code（Anthropic 官方的 AI 编程助手）这个"AI 干活机器"彻底拆开，告诉你它内部是怎么搭起来的——AI 怎么一轮轮自己跟自己对话把活干完、怎么管权限不乱来、怎么在"记忆有限"的情况下还能记住该记的事。

书名"御舆"取自古代造马车的典故：马车里有承载乘客的车厢（舆）、定方向的辕、传力的辐……每个零件各司其职合起来车才能跑。作者用这个比喻说：搭一个 AI Agent 也一样，那个把所有零件托起来、让 AI 真正转起来的"运行时骨架"，就叫 Agent Harness（可以理解成 AI 助手的"底盘 + 神经系统"）。市面上都在教你怎么"用"AI，这本书带你拆开它看里面。

应用场景

想自己做一个 AI Agent 产品的人：你不想只会调 API，想知道一个"生产级"的 AI 助手底层骨架长什么样、有哪些坑，把它当设计参考蓝图。
重度用 Claude Code / Cursor 这类工具的人：搞懂它为什么这么设计，从而把工具用得更狠——知道它什么时候会"压缩记忆"、权限为什么要拦你。
技术管理者 / 架构师：想给团队选型或自研 Agent 框架时，手里有一张"设计决策地图"，知道每个选择的取舍在哪。
研究者 / 想写文章的人：拿它当一份系统性的二手资料，书里有 139 张架构图可以直接引用。

社区评价

起源于国内技术社区 Linux.Do，被以"开源千星"专帖收录推荐，定位是"系统性讲 Agent Harness 设计原理"的文档共建项目。（linux.do 讨论帖）

02

解决了什么痛点

为什么有人愿意花时间写 42 万字？它回应了什么真实问题？

想自己做 AI Agent 的人，普遍卡在这几件事上

第一个痛点：网上全是"怎么用"，没人讲"怎么搭"。你搜 AI Agent，铺天盖地都是"如何写好 Prompt""如何调用 Agent API"这类使用教程。但当你真想自己做一个能干活、能上线的 AI 助手时，你会发现一个尴尬的事实——一个生产级 Agent 系统的"骨架"到底是怎么搭的，几乎查不到任何系统性资料。大家都停在"会用"，没人带你进"会造"。这本书就是冲着这个空白来的。

第二个痛点：自己摸索的 Agent 一到真实场景就散架。很多人照着教程拼出一个 demo，单次对话挺好，可一旦任务变长、要调好几个工具、要管权限、要处理"AI 记忆塞满了怎么办"，整个系统就开始各种崩。问题在于缺少对底层机制的理解：AI 怎么一轮轮自我驱动（对话循环）、权限怎么分阶段拦截、上下文（AI 的"工作记忆"）满了怎么自动压缩。这些都是"魔鬼藏在细节里"的地方，而这本书恰恰逐个拆给你看。

第三个痛点：学到的东西换个框架就白学。市面上 Agent 框架一大堆——LangChain、AutoGen、CrewAI——你深扎一个，换工具就得重学。作者的主张是：与其学某个框架的 API，不如吃透 Claude Code 这个"标杆产品"背后的设计原理，提炼出一套可以迁移到任何框架的心智模型。看懂它"为什么这样设计"，你手里就有了一张不会过期的地图。

御舆：解码 Agent Harness
Claude Code 架构深度剖析

"一器而工聚焉者，车为多。" ——《考工记》

两千年前，造一辆马车是最复杂的系统工程：舆承载乘者，辕定方向，辐传动力，軎辖为约束——每个构件各司其职，合而为一，车方能行。

今天，构建一个 AI Agent 亦是如此：对话循环为辕，工具系统为辐，权限管线为軎辖，而将这一切承载于其上、使智能体真正运转的运行时框架—— Agent Harness——正是那个舆。

古人御舆，驾驭的是天地之间最精密的机械；今人御舆，驾驭的是硅基时代最复杂的智能体系统。本书因此得名 舆书。

大白话

书名叫"御舆"，配了句《考工记》（古代工艺典籍）的话，意思是：一件东西上汇集的工种最多的，就是车。

作者拿"造马车"作比喻：马车里"舆"是车厢（承重）、"辕"是车把（定方向）、"辐"是车轮条（传力）、"軎辖"是固定轮子的销子（约束）。零件各管一摊，合起来车才能跑。

做 AI Agent 同理：对话循环管方向、工具系统管出力、权限管线管约束；而把这一切托起来、让 AI 真正转起来的"运行时骨架"，就叫 Agent Harness。它就是那个最关键的"车厢"。书名由此而来。

04

创新点

同样在讲 AI Agent，它和别的资料有什么不一样？

和常规做法不一样的地方

绝大多数 AI Agent 资料停在"怎么用"的层面——教提示词、教调接口。这本书反其道而行：它选了一个"标杆产品"（Claude Code），把它当成解剖样本，专门讲"为什么这么设计"。它的创新不在某个新技术，而在视角和组织方式：用一套统一的比喻把零散的技术细节串成一个完整的认知体系。

🛞
用"造马车"统一全书叙事

把对话循环、工具、权限这些抽象技术，对应成马车的辕、辐、軎辖。技术书最难的是"让外行也建立整体感"，这个贯穿全书的比喻是个很聪明的内容设计——它降低了认知门槛，也让全书有了记忆点。

🔍
逆向"解剖"而非正向教学

不是"我教你从零搭"，而是"我们一起拆开这个成熟产品看它怎么搭的"。借一个真实跑在生产环境、被海量用户验证过的系统当案例，比凭空讲原理可信得多——相当于不讲菜谱，直接带你进米其林后厨看。

🧭
强调"可迁移"，不绑定框架

市面资料大多绑死某个框架（学了 LangChain 换个工具就废）。这本书刻意提炼"换到哪都通用"的设计原则，并把 139 张架构图做成可复用资产。卖的不是知识点，是一套"看任何 Agent 系统都用得上的脑图"。

📚
体量与体系化做到极致

42 万字、15 章、139 张图、附录做成可查工具手册——在"碎片化短文 + 视频"主导的时代，反而回归"一本厚书系统讲透一件事"。这种密度和完整度本身就是稀缺的，也是它能快速涨到几千 star 的原因。

05

灵感卡片

这本书讲的是技术，但里面几个"设计思想"挪到产品上，意外地好用

灵感 1
"只保存无法推导的信息"——可以变成一条内容/记录的减负原则

书里 AI 记忆系统有条铁律：能从别处推出来的就别存，只存推不出来的。这其实是个万能的"减噪"原则。任何让人填表、记笔记、做归档的产品环节，都可以问一句：这个字段是不是别处已有、能自动带出来？砍掉冗余填写，体验立刻变轻。

灵感 2
"四级渐进压缩"——信息过载时别一刀切，分档处理

AI 内存快满时不是直接清空，而是分四档逐步瘦身（先精简、再合并、最后才大刀阔斧）。这套"渐进降级"思路很适合内容/消息类场景：群消息太多时，先折叠、再摘要、最后才归档，而不是粗暴地"已读 99+"。给用户一个柔和的过载缓冲。

灵感 3
"包工头只派活不干活"——团队/协作类功能的分工模型

书里的协调器模式：一个 AI 只负责拆任务、派活、汇总，绝不亲自下场执行。这条"编排者与执行者严格分离"的约束，放到任何有"管理者 + 一线"结构的产品里都成立——比如代理团队的任务分发，上层只管派单与统计，执行细节完全下放，权责清晰反而更高效。

灵感 4
"先想后做"的 Plan 模式——给用户一个可确认的"预演"

Claude Code 有个模式：动手前先把计划列出来给你看，你点头了才执行。这个"高风险动作先预演、再确认"的交互，对任何不可逆操作都值钱：批量发送、批量改价、批量删除前，先弹一个"我将要做这 X 件事"的清单页，把焦虑和误操作一起消掉。

灵感 5
把"技术大部头"拆成可查的工具手册——内容产品的双层结构

这本书正文讲思想、附录做成速查表（工具清单、术语表、开关表）。这是很高级的内容结构：一层供"系统学"，一层供"用时查"。你做任何知识型/帮助类内容（代理培训手册、运营 SOP）都能借鉴——别只写成一篇长文，配一份能随时跳查的索引，使用率会高得多。

产品化可行性

一个不写代码的 PM，"用"这个项目能得到什么？

💡
先理清一件事：它是"书"，不是"工具"

这个项目没有可安装的软件、没有能调用的功能。它是一本免费在线书。所以"可行性"在这里要换个问法：不是"我能不能用它做出东西"，而是"我读它划不划算、能读懂多少、能带走什么"。

零门槛
获取成本

完全免费、网页直接读，无需安装、注册、付费。打开 lintsinghua.github.io 就能看。

偏高门槛
阅读门槛

面向架构师/工程师写，满是代码和技术术语。对无技术背景的 PM，能吸收的是"设计思想"层，细节会很吃力。

看用途
对你的实用度

想自研 Agent / 带技术团队 → 价值高；只想找产品灵感 → 读前两章 + 各章开头的"为什么"即可。

如果你要读，怎么读最划算

别从头啃，会被代码劝退。建议路线：① 先读第 01 章（建立整体认知）和各章开头那段"为什么这样设计"——这部分是大白话、含金量最高，也最能转化成产品灵感；② 跳过所有代码块和带 <I,O,P> 这种符号的段落，不影响理解大局；③ 把它当"睡前读物 + 灵感库"，而不是"教程"。投入产出最高的方式是只取思想、不啃实现。

⚠️
几个要留意的点

① 它是"二手解读"：作者声明是靠公开文档反推的，不是官方源码，所以细节可能有出入，别当成绝对权威。
② 协议禁止商用：CC BY-NC-SA 意味着你不能拿它的内容去做收费课程/产品，自己学习和团队内参没问题。
③ 会过时：Claude Code 更新很快，书里某些具体数字/机制可能很快和最新版对不上——看思想比看细节保值。

同类资料速览

想搞懂 Claude Code / Agent 架构，除了它还有哪些选择

官方文档 Anthropic 官方的 Claude Code 文档。最权威、最新，但只讲"怎么用"，不讲"为什么这么设计"。适合上手，不适合理解内部。
李博杰博客 《一场泄露看懂 Claude Code：Harness 是让 Agent 干活靠谱的关键》。一篇深度长文，是这本书的"轻量版"——想快速了解核心观点、不想读 42 万字的人，先看这篇。（01.me）
各类逆向分析文 源码泄露后社区涌现了一批"Claude Code 源码逆向"文章/指南，零散、深浅不一。胜在新鲜及时，输在不成体系——这本书相当于把这些零散讨论系统化、结构化了。
框架官方教程 LangChain / AutoGen / CrewAI 等自家文档。教你用具体框架搭 Agent，能直接动手，但绑定单一框架、且不会带你看"标杆产品"的设计取舍。
📌
这本书适合：

想系统、深入理解一个生产级 Agent 到底怎么搭的人——架构师、要带队自研 Agent 的工程负责人、做研究的人。也适合想用一套统一框架建立"Agent 全局观"的技术读者。


不适合：

只想快速上手 Claude Code 的使用者（看官方文档更快）；想找"复制粘贴就能跑"的代码模板的人（这是讲思想的书，不是脚手架）；以及完全无技术背景、只想要结论的人（建议直接看本报告 + 李博杰那篇长文就够了）。
