#Claude Code #AI 编程 #Agent Skills #工程方法论

这是什么？

一位资深工程师（Matt Pocock，TypeScript 圈名人）把自己每天用来跟 AI 编程助手协作的一整套「工作习惯」公开了出来。它不是软件、不是工具，而是一组写给 AI 看的「行为说明书」——每条说明书教 AI 在某个环节该怎么做事：动手前先盘问清楚需求、写代码前先写测试、定期帮你给代码做体检防止越改越乱。

用大白话说：别人卷的是「让 AI 写得更快」，他卷的是「让 AI 像个靠谱的资深同事一样有章法地干活」。

应用场景

动手前对齐需求：你想让 AI 做个功能，但自己也没完全想清楚。先用 /grill-me 让 AI 反过来把你「拷问」一遍，把模糊的想法问到水落石出，再动手——避免做出来发现「这不是我要的」。
给代码做定期体检：AI 改得越快，代码越容易变成一团乱麻。每隔几天跑一次 /improve-codebase-architecture，它生成一份可视化报告告诉你哪里在腐烂、该怎么收拾。
啃硬骨头 Bug：遇到怎么都查不出的疑难杂症或性能问题，用 /diagnosing-bugs 让 AI 按「复现 → 缩小范围 → 提假设 → 加日志 → 修复 → 补测试」的固定套路一步步排查，而不是瞎猜。
把一次聊天变成正式需求文档：你和 AI 聊完一个方案，用 /to-prd 直接把对话整理成 PRD 并发到你的任务系统（GitHub / Linear），再用 /to-issues 拆成一个个可独立认领的小任务。

社区评价

一位用了一周的开发者总结：别一上来就装全部 28 个技能，先用 4 个就够——/grill-with-docs、/tdd、/diagnose、/caveman，需要了再加。（Devtalk 论坛）
同一篇实测里，最被称赞的是 /grill-with-docs——「动手前的盘问 + 自动沉淀项目文档，让后面每一次对话都更省力」。但 /tdd 评价两极：写纯函数好用，做复杂 UI 组件时「需求很难提前用测试写死，反而拖慢节奏」。（Devtalk 论坛）
有第三方评论把它和 GSD / BMAD / Spec-Kit 这类「全流程框架」对比，认为它代表另一种思路：AI 是一个有主见、内化了某种工程文化的团队成员，而不是被流程框死的执行机器。（Tosea.ai 指南）

解决了什么痛点

为什么一个资深工程师要花力气写这套东西？它回应了什么真实问题？

用 AI 写代码的人，几乎都踩过这四个坑

痛点一：「它没做我想要的」。这是软件开发里最经典的失败——你以为对方懂你，结果做出来的东西完全跑偏。AI 时代一模一样：你和 AI 之间有一道「沟通鸿沟」，你扔一句话过去，它脑补一大堆。作者的解法不是写更长的提示词，而是让 AI 反过来盘问你——逼你把没想清楚的地方在动手前就想清楚。

痛点二：「它太啰嗦了」。AI 被丢进一个陌生项目，不懂这个项目里的「行话」（比如你们内部管某个东西叫什么），只能用 20 个词去描述本该 1 个词说清的事。结果就是又慢又啰嗦，每次还得重新理解一遍。作者的解法是给项目建一份「术语词典」（CONTEXT.md），让 AI 和你说同一套语言——他说这「可能是整个项目里最酷的一招」。

痛点三：「代码跑不起来 / 不对」。就算需求对齐了，AI 照样能产出一堆垃圾。根因是它「闭着眼睛飞」——没有反馈，不知道自己写的东西到底行不行。解法是给它装上反馈回路：类型检查、能看浏览器、自动化测试，尤其是「先写一个会失败的测试、再把它修绿」这种红绿循环。

痛点四：「我们搭出了一坨烂泥」。AI 让写代码快了 10 倍，也让代码变烂快了 10 倍——「软件熵增」被极大加速，项目以前所未有的速度变得又复杂又改不动。作者认为唯一的解药是一种「反潮流」的态度：每天都认真对待代码的设计，并提供工具定期把已经腐烂的代码救回来。

## Reference

These split on one axis — who can invoke them. User-invoked skills are reachable only when you type them (e.g. /grill-me); their job is to orchestrate. Model-invoked skills can be invoked by you or reached for automatically by the agent when the task fits; they hold the reusable discipline. A user-invoked skill may invoke model-invoked skills, but never another user-invoked one.

## 参考清单

04

创新点

和同类方案比，它有什么不一样？

和常规做法不一样的地方

市面上多数「AI 编程方法」走的是「接管整个流程」的路子（GSD、BMAD、Spec-Kit 等）——你把方向盘交出去，它带着你走完一套固定流水线。Matt 反着来：他把流程拆成一颗颗小而独立的「行为习惯」，你想用哪颗用哪颗，方向盘始终在你手里。它的创新不在「某个炫技功能」，而在一整套「怎么和 AI 协作才靠谱」的工程哲学，并把它落成了能直接装进项目的东西。

🔄
让 AI 反过来盘问你

绝大多数人是「我说、AI 做」。他把它倒过来：动手前先让 AI 连环追问，把你自己都没想清的需求逼出来。一个简单的方向反转，直接命中「做出来不是我要的」这个最大痛点。

📖
给项目建一本「行话词典」

CONTEXT.md——让 AI 和你说同一套术语。作者称这「可能是整个 repo 里最酷的一招」：不仅让 AI 不啰嗦，还顺带让命名一致、代码好导航、AI 思考更省 token。

🧱
小巧、可改、可拼搭

每个 skill 都被刻意做小，鼓励你「拆开改成自己的」。这和那些「黑箱式、改不动」的大框架是相反的设计哲学——你永远没失去对流程的控制。

🩺
把「代码体检」变成日常

承认了一个反直觉的事实：AI 让写代码快 10 倍，也让代码烂得快 10 倍。于是配了把「定期清淤」工具，主张每隔几天给代码库做一次架构体检——把「治理腐烂」纳入常规节奏。

🎚️
「谁能触发」两档分层

把 skill 分成「用户手动触发」（负责统筹编排）和「模型自动触发」（承载可复用纪律）两类，并立了规矩：手动的可以调自动的，但绝不互相调用。一套清晰的「权限/职责」设计，避免 AI 乱套娃。

05

灵感卡片

看完这个项目我想到的产品点子——不一定用它的代码

灵感 1
「先盘问、再做单」——给运营的提需求助手

这个项目最核心的一招是「动手前让 AI 反过来追问你」。搬到咱们的做单场景：运营想发一款商品，但群文案/图怎么打、卖点是什么往往没想清。可以做一个「做单前盘问」流程——AI 先问几个关键问题（这单主打什么人群？强调低价还是稀缺？配几张图？），把模糊的想法问清楚再生成内容。能直接降低「做出来的单不出单」的概率。

灵感 2
给团队建一本「行话词典」

他给项目建 CONTEXT.md，让 AI 听得懂内部黑话。咱们团队和代理之间其实也有大量黑话（转链、做单、私域……还有各种内部叫法）。可以沉淀一份「团队术语表」，无论是培训新代理、还是以后让 AI 帮运营写文案/答疑，都能让对方一上来就听懂咱们的语言，少解释一遍。

灵感 3
「定期体检」式的运营健康巡检

他主张每隔几天给代码库做一次「架构体检」，主动发现腐烂。这个「定期主动巡检」的思路可以搬到运营：每周自动扫一遍各群的活跃度、出单率、沉默用户，生成一份「哪几个群在腐烂、建议怎么救」的可视化报告。把「等出问题才救火」变成「定期主动发现」。

灵感 4
「小而可拼搭」的运营 SOP 卡片

他不做「一键全自动」的大框架，而是拆成一颗颗小习惯、随用随取。咱们给代理的运营手册也可以这样：别写成一本几十页没人看的大文档，拆成一张张「单点 SOP 卡片」（怎么发新品、怎么追单、怎么处理售后），代理按当下需要抽一张用。小、好改、好传播。

灵感 5
把一段聊天直接变成「需求卡」

他的 /to-prd 能把一次对话直接整理成正式需求并拆成小任务。你作为 PM，平时和我（或团队）聊产品聊得很碎。可以做个「聊完即沉淀」的小习惯/工具：一段讨论结束，一键把它整理成结构化的需求卡（背景/痛点/方案/下一步），免得好想法聊完就散在聊天记录里。

06

产品化可行性

一个不会写代码的 PM，能不能用这个项目做出东西来？

低门槛
上手难度

本质是给 AI 看的「说明书」，不是要部署的软件。装上 Claude Code 一类工具后一行命令就能安装，再用大白话调用，几乎纯配置。

中等
依赖链

前提是你得先在电脑上用着一个 coding agent（Claude Code / Codex 等），并大致能在终端里跑命令。对纯做单运营来说，这一步是门槛。

低门槛
Vibe Coding 适配度

它本身就是为「跟 AI 对话式开发」而生的，用法就是聊天。会用 Claude Code 的人能立刻上手；它甚至是用来「治」瞎写的解药。

如果你要做，大概需要什么

对你个人——尤其是你也在做独立小产品/工具时——这套东西非常值得直接用：你需要的只是一台装好 Claude Code 的电脑、跟着 30 秒安装步骤走一遍、然后像聊天一样调用。不花钱（MIT 协议，免费随便用，只是作者会引导你订阅他的 newsletter）。真正的成本是「学会这套工作习惯」——大概花一两天，挑 3～4 个核心 skill 先用起来，而不是一口气全装。

但如果目标是「把它做成给返利团队代理用的东西」，那它不能直接拿来用——它是给「会用编程工具的工程师」的，不是给「在手机微信里做单的运营」的。对你的团队场景，它的价值在方法论层面的启发（见灵感卡片），而不是直接落地的工具。

⚠️
注意

① 这是一个人的个人习惯集，更新很快、风格很「私人」，别指望它像产品一样稳定；② 社区实测共识是「别贪多」——28 个 skill 全装反而乱，先用 4 个；③ 它和某些「全流程框架」哲学相反，如果你团队已经在用 BMAD 那类东西，两者混用可能打架；④ 它的价值高度依赖「人本身懂工程」——你照着用能提效，但它替代不了工程判断。

07

同类项目速览

它在整个生态里处于什么位置

Spec-Kit GitHub 官方的「规格驱动开发」工具包。走「重流程」路线——先写详尽规格，再让 AI 照着实现。和本项目正是 README 里点名对比的对象：管得多、控制感强，但灵活度低。适合流程严谨的团队。
BMAD-METHOD 把 AI 编成「产品经理/架构师/开发」等多角色协作的一整套敏捷框架，社区很热。同样是「框架接管流程」派。本项目作者明确表示这类方法「夺走了你的控制权」，自己反其道而行。
GSD（Get Sh*t Done） 另一类「全流程托管」方法，强调端到端把活干完。和 Spec-Kit/BMAD 同属「重框架」阵营，是本项目刻意要区别开的参照系。
官方 / 社区 Skills 市场 Anthropic 自带的 skills，以及各类 awesome-claude-skills 合集、claudepluginhub 等。胜在数量多、覆盖广，但大多是零散单点；本项目胜在「成体系、有工程哲学贯穿、出自一位名人之手」。
📌
这个项目适合：

已经在用 Claude Code / Codex、有一定工程素养、想「保留控制权」又想提质提效的开发者；以及像你这样、自己也做独立小产品、希望养成一套靠谱 AI 协作习惯的人。


不适合：

① 完全不碰命令行、只在手机微信里做单的运营——它不是给他们的工具；② 想要「一键全自动、不用动脑」的人——它恰恰要求你深度参与、保留判断；③ 已重度绑定某个「全流程框架」且不想改习惯的团队。
