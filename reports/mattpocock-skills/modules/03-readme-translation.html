「没有人能确切知道自己想要什么。」

The Problem. The most common failure mode in software development is misalignment. You think the dev knows what you want. Then you see what they've built - and you realize it didn't understand you at all.

在 AI 时代也一样。你和 agent 之间存在沟通鸿沟。解法是一场 grilling session（盘问环节）——让 agent 就你要做的东西向你提出详细的问题。

### #2: The Agent Is Way Too Verbose

With a ubiquitous language, conversations among developers and expressions of the code are all derived from the same domain model.
— Eric Evans, Domain-Driven-Design
中文

### #2：Agent 太啰嗦了

有了一套「通用语言」（ubiquitous language），开发者之间的对话与代码的表达，就都源自同一个领域模型（domain model）。
—— Eric Evans，《领域驱动设计》
原文

The Problem: At the start of a project, devs and the people they're building the software for (the domain experts) are usually speaking different languages.

I felt the same tension with my agents. Agents are usually dropped into a project and asked to figure out the jargon as they go. So they use 20 words where 1 will do.

中文

问题：在项目初期，开发者和他们服务的对象（领域专家 domain experts）通常说着不同的语言。

我和我的 agent 之间也有同样的张力。Agent 通常是被「空降」进一个项目，被要求边做边搞懂那些行话。于是它会用 20 个词去说本来 1 个词就够的事。

The Fix for this is a shared language. It's a document that helps agents decode the jargon used in the project.

中文

这个问题的解法是一套共享语言。它是一份文档，帮助 agent 解码项目里用到的行话。

### #3: The Code Doesn't Work

"Always take small, deliberate steps. The rate of feedback is your speed limit. Never take on a task that's too big."
— David Thomas & Andrew Hunt, The Pragmatic Programmer
中文

### #3：代码跑不通

「永远迈小而审慎的步子。反馈的速率就是你的速度上限。永远别接一个太大的任务。」
—— David Thomas & Andrew Hunt，《程序员修炼之道》
原文

The Problem: Let's say that you and the agent are aligned on what to build. What happens when the agent still produces crap?

It's time to look at your feedback loops. Without feedback on how the code it produces actually runs, the agent will be flying blind.

中文

问题：假设你和 agent 已经对齐了要做什么。可如果 agent 仍然产出一堆垃圾呢？

那就该检查你的反馈回路（feedback loops）了。如果拿不到「它写的代码实际跑起来怎么样」的反馈，agent 就是在闭眼飞行。

原文

The Fix: You need the usual tranche of feedback loops: static types, browser access, and automated tests.

For automated tests, a red-green-refactor loop is critical. This is where the agent writes a failing test first, then fixes the test. This helps give the agent a consistent level of feedback that results in far better code.

中文

解法：你需要那一整套常规的反馈回路：静态类型（static types）、浏览器访问能力，以及自动化测试。

对自动化测试来说，「红-绿-重构」（red-green-refactor）循环至关重要。也就是 agent 先写一个会失败的测试，然后再去把它修通过。这能给 agent 稳定的反馈，从而写出好得多的代码。

原文

I've built a /tdd skill you can slot into any project. It encourages red-green-refactor and gives the agent plenty of guidance on what makes good and bad tests.

For debugging, I've also built a /diagnosing-bugs skill that wraps best debugging practices into a simple loop.

中文

我做了一个 /tdd skill，你可以把它塞进任何项目。它鼓励红-绿-重构，并就「什么是好测试、什么是坏测试」给 agent 大量指引。

关于调试，我还做了一个 /diagnosing-bugs skill，把最佳调试实践包装成一个简单的循环。

## Reference

These split on one axis — who can invoke them. User-invoked skills are reachable only when you type them (e.g. /grill-me); their job is to orchestrate. Model-invoked skills can be invoked by you or reached for automatically by the agent when the task fits; they hold the reusable discipline. A user-invoked skill may invoke model-invoked skills, but never another user-invoked one.

中文

## 参考清单

这些 skills 按一条轴划分——谁能调用它们。User-invoked（用户触发）的 skills 只有当你亲手输入（比如 /grill-me）才会触发；它们的职责是「编排/统筹」。Model-invoked（模型触发）的 skills 可以由你触发，也可以在任务合适时由 agent 自动调用；它们承载的是「可复用的纪律/方法」。一个用户触发的 skill 可以调用模型触发的 skills，但绝不会去调用另一个用户触发的 skill。

原文
